# Structured Recipe Discovery Pattern

## Purpose

This note captures a public-safe pattern for treating recipe prompts as a structured discovery vertical instead of a generic chat answer. It intentionally omits private source code, private URLs, account data, screenshots, routing internals, tokens, infrastructure names, and production configuration.

## Product Pattern

Food and recipe prompts should return a recipe experience, not a plain model blob.

A useful recipe answer can include:

- a selected top recipe with a source link
- a clear reason why it was selected
- rating and review count only when sourced
- prep time, cook time, servings, and image only when available
- separated ingredients and cooking steps
- tips or substitutions
- alternate recipe cards
- an action to open the original recipe

The assistant can still use a language model to summarize, format, or explain the result, but search and source extraction should run first for recipe intent.

## Source and Provenance Rules

Do not invent source metadata.

Public-safe recipe discovery should distinguish between:

- sourced fields from structured data or reliable public metadata
- search-result metadata such as title, snippet, and canonical URL
- assistant-authored fallback guidance when a source does not expose reusable recipe details
- unavailable fields

If a public page is a roundup or listicle that does not expose reusable ingredients or steps, say that plainly. A product can offer a clearly labeled starter recipe, but it should not pretend those ingredients or steps came from the selected source.

## Image Policy

Recipe cards benefit from food imagery, but image handling needs policy.

Safe options:

- use a source image only when current policy allows it
- otherwise generate a private illustrative food image
- label generated imagery clearly, such as "Illustrative image generated by the assistant"
- keep fallback image prompts food-only, without people, text, account details, or private context

This keeps the recipe card visual without silently hotlinking, scraping, or leaking private data.

## Intent Guardrails

Recipe intent should require food or cooking evidence.

Positive examples:

- "French toast recipe"
- "best chili recipe"
- "how do I make chicken enchiladas"
- "easy dinner recipe"
- "dinner ideas with chicken and rice"

Negative examples:

- "how can I make money quickly"
- "write a Python recipe scraper"
- "generate an image of French toast"
- "recipe for success"
- "make a bash script"

The key lesson is that broad "how do I make..." phrasing is not enough by itself. The prompt needs a food, cooking, meal, or dish signal.

## UI Contract

A recipe card should be rendered as product UI:

- title, source, and top-pick reason are prominent
- ingredients and steps are visually separated
- source/rating/time chips are conditional on real data
- alternates are compact
- details or diagnostics are collapsed by default
- Basic users do not see raw model, route, hardware, provider, or debug internals
- mobile layout has stable image dimensions and no horizontal overflow

Raw Markdown headings such as `### Ingredients` should not leak as visible command-line-style text when the product has a recipe card component.

## Release Gates

Reusable gates for recipe discovery:

- recipe prompts invoke the recipe vertical
- non-recipe prompts do not
- at least one source link exists for recipe prompts
- rating/review fields are shown only when sourced
- ingredients and steps are separated
- a hero image exists from an allowed source or a labeled generated fallback
- generated fallback imagery is labeled illustrative
- no raw Markdown or diagnostics dominate the first fold
- Basic DOM has no raw internal model or route details
- mobile screenshots have no overflow
- code, document, photo, Learning, image, and long-generation paths do not regress

The reusable architecture pattern is "structured vertical first, model polish second": retrieval, provenance, and UI contracts should win over generic model completion for domains where users expect sourced, inspectable results.
